# v1.5 Release Checklist

This checklist is the fail-closed bar for MUTX v1.5.

| Area | Owner |
| --- | --- |
| Backend and contracts | `area:api` |
| Dashboard and browser UX | `area:web` |

## Gate

| Desktop cockpit smoke | `area:desktop` | `npm run test:e2e:release` passes the desktop cockpit scenarios against the standalone build |

| Helm template render | `area:infra` | `helm template mutx infrastructure/helm/mutx/` renders without errors |

| Public release summary | `area:web` | `https://mutx.dev/releases` reflects the current desktop artifact set and links |
| Surface positioning | `area:docs` | preview/stable language in docs matches the actual gated surface set |

## Positioning Rule

Promote `app.mutx.dev/dashboard` to **Supported** only when all checks above pass on the release candidate machine, including signed packaging, notarization, checksum publication, and launch validation.

Keep preview-labeled control routes and partially implemented backend surfaces out of primary stable navigation until their live contracts are complete.

## Launch Order

1. Confirm the notarized macOS assets exist on the `v1.5.0` GitHub release.
2. Confirm `docs/releases/v1.5.md` is merged, `https://docs.mutx.dev/docs/v1.5` is live, and `https://mutx.dev/releases` reflects the same release.
3. Verify RBAC enforcement on `/audit/*`, `/security/*`, `/policies/*`, and `/approvals/*` routes.
4. Verify OIDC token validation with the configured IdP (skip only when no external IdP is in scope).
5. Verify `/v1/pico/progress` matches the generated OpenAPI spec and frontend types.
6. Run `helm lint infrastructure/helm/mutx/` and `helm template` to validate the chart.
7. Run the Railway production-promotion workflow for the frontend and backend services.
8. Run `bash scripts/verify-production-release.sh`.
9. Verify the public download, release, auth, dashboard, API, docs, and Kubernetes surfaces point to the supported v1.5 surfaces.
