# MUTX v1.1

MUTX v1.1 is the first release note that matches the actual scope of the repo.

This is not just a site refresh or a landing-page pass. The repository now ships an operator stack that spans:

- a FastAPI control plane mounted under `/v1/*`
- a Next.js public site plus browser control-plane demo
- a Python CLI with first-party Textual TUI
- a Python SDK
- a local Docker-backed dev and demo stack
- assistant-first install, onboarding, runtime tracking, and deployment flows

Repository snapshot for this release:

- 1,352 commits
- 741 tracked files
- 25 FastAPI route modules
- 86 CLI/TUI files
- 246 test files

## What v1.1 actually delivers

### 1. A real assistant-first operator path

MUTX now behaves like an operator shell, not a disconnected pile of surfaces.

The core path is:

1. install MUTX with `curl -fsSL https://mutx.dev/install.sh | bash`
2. choose hosted or local
3. authenticate or bootstrap the operator state
4. resolve the assistant runtime
5. deploy the starter personal assistant
6. inspect health, sessions, and control-plane state from the CLI, TUI, or dashboard

The command surface around that path now includes:

- `mutx setup hosted`
- `mutx setup local`
- `mutx doctor`
- `mutx assistant overview`
- `mutx assistant sessions`
- `mutx runtime list`
- `mutx runtime inspect openclaw`
- `mutx runtime open openclaw --surface tui`
- `mutx tui`

### 2. OpenClaw as the first real runtime provider

OpenClaw is now wired in as a tracked runtime provider inside MUTX rather than an undocumented external side quest.

MUTX can now:

- detect an existing OpenClaw installation
- import that runtime into `~/.mutx/providers/openclaw`
- keep the upstream OpenClaw home where it already lives
- preserve local gateway credentials on the local machine
- bind a dedicated assistant runtime instead of reusing `main`
- hand off to `openclaw tui` or `openclaw configure` and return to the MUTX shell

The tracked provider story includes:

- runtime manifests
- assistant bindings
- wizard state
- last-seen snapshot sync for hosted surfaces
- local-truth overlays in CLI and TUI

### 3. Hosted and local setup that make operational sense

The hosted lane is the recommended path and now behaves like it.

The local lane is no longer a fake contributor-only option. It can provision a managed localhost stack under `~/.mutx/runtime/local-control`, bring up the local control plane, and continue through the same assistant/runtime flow.

Key setup improvements:

- narrower, safer installer rendering
- wizard continuity through setup instead of dropping to raw prompts too early
- better stale-binary recovery
- OpenClaw import versus install/repair decisions when an existing runtime is found
- cleaner privacy copy around local keys and upstream home directories

### 4. Honest runtime visibility

The hosted API no longer pretends it can observe operator-local OpenClaw state directly.

Instead, MUTX now distinguishes between:

- live local truth available in the operator environment
- last-seen synced runtime state stored for hosted control-plane views
- runtime states that require the client to inspect local infrastructure

That makes the CLI, TUI, and dashboard agree more often, and lie less often.

### 5. A usable browser and dashboard story

The landing page, quickstart surfaces, and browser control-plane demo were tightened so the product story now matches the actual workflow:

- MUTX owns the shell
- OpenClaw is shown as the first enabled provider
- the quickstart is simpler
- the setup and dashboard surfaces are less hand-wavy

This release also keeps the public browser demo in a more coherent relationship with the operator-first CLI and TUI flows.

### 6. Local Docker/demo flow that actually boots

One of the ugliest gaps in the repo was the local proof path: the stack looked available on paper, but real boot validation could still fail for avoidable reasons.

v1.1 closes the biggest bootstrap failures:

- compose project names are isolated instead of colliding with stale containers
- the demo validator now tears down or reuses the correct project scope
- migrations run before API readiness
- readiness checks use `/ready`
- the sync database path uses `psycopg`
- the `user_settings` migration now matches the UUID user schema
- local dev JWT defaults are valid enough to boot the stack cleanly

The result is that `npm run demo:validate` now succeeds as a real proof path instead of a flaky claim.

## Why this release matters

There are already too many agent projects that can generate output but cannot explain how to operate, inspect, recover, or deploy the system around the model runtime.

MUTX is trying to be the control plane around those runtimes:

- durable assistant and agent identity
- deployment and runtime records
- CLI, TUI, browser, and API surfaces that talk about the same objects
- local-runtime honesty instead of fake hosted omniscience

v1.1 is the point where that story becomes materially more believable from the repo itself.

## Packaging and distribution

This release also tightens the distribution story:

- CLI package version: `0.2.1`
- Homebrew release tag: `cli-v0.2.1`
- main repo release tag: `v1.1`

The Homebrew tap is updated separately to point at the matching CLI tarball and checksum so `brew install mutx` and the guided installer stay aligned.

The repo is already broader than the old release notes admitted. v1.1 is meant to fix that and present the project as it actually exists today: a control-plane-first operator system with a real install story, a real runtime provider flow, and a much stronger local proof path.
