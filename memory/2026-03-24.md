## OpenClaw Capabilities Audit — Key Learnings

Today was an OpenClaw capabilities audit. Here's what we learned:

### What OpenClaw Can Do

- **Spawn subagents** via `sessions_spawn` — isolated sessions with their own context, can run in parallel
- **Spawn ACP harness sessions** (Codex, Claude Code, Pi) via `sessions_spawn(runtime="acp")` — thread-bound, persistent
- **Exec on remote macOS node** — `exec(host="node", node="MacBook Pro")` for Apple-only skills (Things, Notes, etc.)
- **File operations** — read/write/edit with full workspace access
- **Browser automation** — snapshot/act on web pages (login-aware, supports profiles)
- **Web fetch** — pull readable content from URLs
- **Skills system** — `SKILL.md` files with specialized instructions; clawhub for installing new ones
- **Discord integration** — send/react/thread/reply via `message(channel="discord")`
- **Tmux integration** — control remote tmux sessions for interactive CLIs
- **Canvas** — render/eval JavaScript, capture snapshots
- **TTS** — text-to-speech via ElevenLabs
- **GitHub CLI** — `gh` for issues, PRs, CI, code review
- **1Password, Apple Notes/Reminders, Things 3, Bear** — platform-specific integrations

### What We Learned About Memory System

- **MEMORY.md** is loaded in main session (direct chats) but NOT in shared/group contexts (security)
- **Daily notes** (`memory/YYYY-MM-DD.md`) are raw logs — what happened, decisions, errors
- **State files** (`mutx-fleet-state.md`, `autonomy-queue.json`) are the operational truth between sessions
- **Worker state** (`workspace-x/worker_state.json`) tracks X agent continuity
- State-first workflow is critical: read state → decide → act → update state

### What We Were Doing Wrong (X / Social)

1. **Volume spam** — posting too frequently, too many posts per session
2. **Thread fragmentation** — starting threads without finishing them, losing context
3. **No quality gates** — posting without reviewing first, letting workers post without supervision
4. **Engagement farming** — prioritizing metrics over signal
5. **X cron workers disabled** — legacy cron shape broken since March 17-18

## What We Got Wrong — Honest Post-Mortem

### X / Social (pre-March-24)

- **Volume chasing** — More posts ≠ more followers. We were posting to hit activity metrics, not to build signal.
- **Thread abandonment** — Started threads, lost them, never finished. Audience remembers.
- **Engagement bait** — Reactive posting instead of proactive thought leadership.
- **No quality gates** — Workers posting without human review. Mistake.
- **Cascade failure** — When rate limits hit, X workers didn't back off gracefully; they spam-retried.

### Fleet (pre-March-18)

- **52 workers** — This was obviously too many. The cascade was predictable.
- **No state continuity** — Isolated sessions with no memory of what happened last run.
- **Fake validation** — Claiming `npm run build` passed when it hadn't (no `node_modules`).
- **Direct-to-main without monitoring** — Pushed to main without watching post-merge CI.

### What We're Doing Right

- **Discipline** — 3-worker max, rate limit backoff, state-first workflow
- **Ship clean** — One green PR > ten broken ones
- **Healer before shipper** — Never try to merge a dirty PR
- **Direct-to-main for internal** — Fast iteration without PR overhead for refactors
- **Quality over volume on X** — (as of today)
