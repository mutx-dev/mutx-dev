# README

## MUTX

MUTX is an open-source control plane for deploying, operating, and governing AI agents.

It helps teams turn agent demos into systems they can trust, operate, and ship.

Full docs live at `docs.mutx.dev`.

{% embed url="https://docs.mutx.dev" %}

### What MUTX covers

* auth and user identity
* agent records and lifecycle actions
* deployment records and lifecycle actions
* API keys and webhook flows
* health, readiness, logs, and metrics
* API, CLI, SDK, and web surfaces that stay aligned

### Why it exists

The hard part of agent software is not token generation.

The hard part is making agents reproducible, observable, governable, and production-safe.

MUTX focuses on that layer.

Most agent projects do not fail because the model is weak.

They fail because ownership is unclear, deployments are implicit, secrets are bolted on late, and no interface agrees with the others.

MUTX exists to make that operational layer explicit.

### What you get in this repo

This repo is the current source of truth for the MUTX platform surface.

It includes:

* a FastAPI control plane
* a web surface for product and operator flows
* a CLI for auth, status, and agent workflows
* a Python SDK for programmatic access
* deployment and infrastructure assets
* current-state documentation that tracks the code closely

### Current repo state

* The API already exposes real auth, agent, deployment, webhook, health, and readiness routes.
* The CLI and SDK are usable, but still being aligned with the live contract.
* The web surface is strong on positioning. The app surface is still early.
* The docs favor current-state honesty over aspirational claims.

If code and docs disagree, trust the code and update the nearest doc.

For the most honest snapshot of what works today, see [Project Status](docs/project-status.md).

### Quickstart

Clone the repo and install dependencies:

```bash
git clone https://github.com/fortunexbt/mutx.dev.git
cd mutx.dev

npm install
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
pip install -e ".[dev]"
cp .env.example .env
```

Set your local database URL in `.env`:

```bash
DATABASE_URL=postgresql://mutx:mutx_password@localhost:5432/mutx
```

Start the local services and API:

```bash
docker compose -f infrastructure/docker/docker-compose.yml up -d postgres redis
uvicorn src.api.main:app --reload --port 8000
```

Start the web app in another terminal:

```bash
npm run dev
```

Verify the local API:

```bash
curl http://localhost:8000/health
curl http://localhost:8000/ready
```

Create a local user:

```bash
curl -X POST http://localhost:8000/auth/register \
  -H "Content-Type: application/json" \
  -d '{"email":"you@example.com","name":"You","password":"StrongPass1!"}'
```

Log in:

```bash
curl -X POST http://localhost:8000/auth/login \
  -H "Content-Type: application/json" \
  -d '{"email":"you@example.com","password":"StrongPass1!"}'
```

Create an agent:

```bash
curl -X POST http://localhost:8000/agents \
  -H "Authorization: Bearer YOUR_ACCESS_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "name":"My First Agent",
    "description":"Local test agent",
    "config":"{\"model\":\"gpt-4\"}"
  }'
```

Deploy it:

```bash
curl -X POST http://localhost:8000/agents/YOUR_AGENT_ID/deploy
```

If you want the full local flow, use [Quickstart](docs/deployment/quickstart.md).

### Principles

MUTX is being built around a few simple rules:

* current-state honesty beats polished fiction
* explicit control beats hidden magic
* small surfaces with strong guarantees beat wide, vague platforms
* docs, API, CLI, SDK, and product surface should tell the same story
* operators should be able to answer who owns a resource, what changed, what failed, and what to do next
