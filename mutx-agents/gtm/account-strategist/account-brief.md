# Account Brief — GitHub Actions / Agentic Workflows

## Account choice
- **Target account:** GitHub Actions / Agentic Workflows
- **Why this account:** the freshest outside-in signal names GitHub as the clearest proof of the market shift toward governed execution: secure defaults, policy controls, scoped credentials, enforceable network boundaries, constrained outputs, and comprehensive logging.

## Working thesis
GitHub is not just a signal source; it is the cleanest public account-shaped proxy for the exact buyer pain MUTX should solve: teams want agentic execution that is safe, auditable, and bounded by policy.

## Stakeholder map
- **Champion:** platform or workflow lead responsible for GitHub Actions or agentic workflow safety; feels the gap between “agents can run” and “agents can run safely.”
- **Operator user:** CI/workflow engineer or platform operator who owns runners, logs, approvals, and boundary enforcement.
- **Buyer:** engineering/platform leadership responsible for secure defaults, governance posture, and rollout confidence.

## Current pain to anchor on
- secure defaults need to be enforced, not described
- policy controls need to be visible to operators
- scoped credentials and approval boundaries need audit trails
- network boundaries and constrained outputs need runtime evidence
- observability needs to show behavior, not just token counts

## MUTX wedge
MUTX should be framed as the control plane that helps teams **deploy, operate, observe, and govern agentic workflows like production infrastructure**.

## Proof boundary
Do **not** claim dashboard completeness or dashboard-first governance. The defensible proof today is:
- supported `/dashboard` operator lane
- real CLI governance path
- runtime inspection and truthfulness around supported vs preview surfaces

## Design-partner path
1. Start with one governed workflow or runner boundary problem.
2. Show MUTX as the layer that gives policy, approvals, and runtime visibility without lying about UI gaps.
3. Expand from a single workflow proof into fleet-level governance, traces, and auditability.
4. Only widen once the account agrees the control-plane framing matches the real operating problem.

## Expansion logic
Once the first workflow is credible, the next expansion is not “more agent demos.” It is:
- broader policy coverage
- more runtime surfaces
- approval visibility
- audit and trace retention
- support for additional teams with the same governance posture

## What to say in outreach / briefings
- governed execution, not generic observability
- secure defaults, policy controls, scoped credentials, network boundaries
- auditability and runtime truth over demo polish
- operator proof over promise
