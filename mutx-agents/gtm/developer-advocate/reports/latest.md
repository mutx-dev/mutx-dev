- Highest-leverage move: turn the truthful `Personal Assistant` install → deploy → inspect → restart loop into the canonical proof asset. That is the cleanest way to show MUTX as an operator control plane, not another agent builder.
- Stronger proof angle: a 15-minute operator walkthrough that starts with `curl -fsSL https://mutx.dev/install.sh | bash`, runs `mutx setup hosted`, verifies `mutx assistant overview` plus `mutx runtime inspect openclaw`, creates a deployment through `POST /v1/deployments`, reads lifecycle history from `GET /v1/deployments/{deployment_id}/events`, and shows the same state in `app.mutx.dev/dashboard` via the `/api/dashboard/overview` contract.
- Why this is safe to tell now: quickstart, CLI setup, dashboard overview, deployment create, deployment events, logs, metrics, and restart are all repo-backed and tested. It stays inside the supported `/dashboard` lane and avoids blocked claims around scheduler, RAG, monitoring self-heal, or runtime magic.
- Exact drift to keep out of the asset: API docs promise deployment versions and rollback, but the SDK `Deployments` client currently stops at logs/metrics and exposes no versions/rollback helpers. CLI docs only advertise the newer `mutx deployment list/create` surface, while richer lifecycle actions still live under legacy `mutx deploy` commands. Until `issue-117` closes, anchor the tutorial on create/list/events/restart instead of rollback/version history.
- Recommendation: ship one truthful deploy-and-operate proof asset around this loop before spending more cycles on monitoring or governance narratives.
